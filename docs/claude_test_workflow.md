# Claude 实测工作流

这是一份测试执行文档，用于说明在 Claude 里怎么测、怎么记、怎么回改。

这份文档只回答一件事：

- 把生成出来的 `SKILL.md` 放进 Claude 之后，应该怎么测、怎么记、怎么回改

它不参与 Skill 生成，也不应该写回 `SKILL.md` 本体。

## 一句话流程

1. 固定一个角色和一个关系身份
2. 用固定测试场景在 Claude 里实际聊天
3. 发现“不像”的地方后，手动记录到测试日志
4. 按问题类型回改对应 Prompt
5. 用同一组测试场景再测一轮

## 先固定测试范围

每次只测一组：

- 一个角色
- 一个关系身份
- 一版 Skill

不要把多个角色、多个身份、多个版本混在同一轮里，不然后面很难判断到底是哪一层出了问题。

## 推荐测试顺序

1. 先从 [eval_cases.md](eval_cases.md) 里挑 5 到 6 个高频场景
2. 把同一份 `SKILL.md` 放进 Claude
3. 在 Claude 里按同样顺序测试这些输入
4. 每测完一条，就把结果记到 [eval_log_template.md](eval_log_template.md)

## 在 Claude 里看什么

每条输出至少看这 4 件事：

1. 像不像这个角色
2. 像不像这个关系身份
3. 像不像真人对话
4. 有没有乱编设定

如果你在 Claude 里说“这里不像”，这不会自动记录下来。
你需要手动把这次结果记进测试日志。

## 最小记录格式

每条至少记这 4 项：

```text
场景类型：
用户输入：
Claude 输出：
问题判断：
```

推荐再补：

```text
结果：通过 / 勉强可用 / 不通过
问题标签：
修改建议：
```

## “这里不像”之后怎么记

最简单的做法是直接记成：

```text
场景类型：用户疲惫
用户输入：今天真的太累了，不想动了。
Claude 输出：……
结果：勉强可用
问题标签：太模板 / 不像角色
修改建议：先加强情绪承接，再收一收提醒感
```

关键点是：

- 记录原始输入
- 记录 Claude 原话
- 不要只写“感觉不太像”
- 最好顺手标一个问题类型

## 记录完之后怎么回改

先看 [issue_mapping.md](issue_mapping.md)：

- 不像角色：
  优先改 `persona_*`
- 设定事实不对：
  优先改 `lore_*`
- 身份不成立：
  优先改 `relationship_builder.md`
- 某个身份下场景不自然：
  优先改 `relationship_scene_builder.md`
- 太模板 / 不像真人：
  优先改 `custom_builder.md`

## 最重要的原则

- Claude 里的“不像”不会自动写回项目
- 测试记录只放在 `docs/`
- 不要把测试结论、问题标签、修改建议写进 Skill 本体
- 每次只改一层或一类问题，再回归测试
